> Retrieval-Augmented Generation 检索增强生成

## RAG 如何工作

RAG 的工作流程正如其名，分为三个阶段：

### 检索 (Retrieval)

当你提出问题 例如：“公司明年的放假安排是什么？”，系统不会直接把问题丢给 AI。

系统会先去你预设的外部知识库（比如公司的文档库、向量数据库）里进行搜索，找到与问题最相关的几段文档。

### 增强 (Augmented)

系统将你原来的问题和刚才搜到的相关文档（上下文）拼接在一起，形成一个更丰富的新提示词（Prompt）。

类似于：“请根据以下公司文档：[文档内容...]，回答用户的问题：[公司明年的放假安排是什么？]”

### 生成 (Generation)

最后，将这个“增强版”的提示词发送给大模型。大模型结合搜索到的事实，生成最终的答案。

## 为什么需要 RAG

RAG 完美解决了大语言模型的三个“硬伤”：

- **解决“幻觉”问题 (Accuracy)**： 大模型有时会编造事实。RAG 强制模型基于提供的参考资料回答，大大降低了胡编乱造的概率，让回答有据可依。

- **解决“时效性”问题 (Freshness)**： 训练一个大模型（如 GPT-4）需要几个月时间和巨额成本，知识更新很慢。使用 RAG，你只需要更新外部数据库（比如今天的具体新闻），模型就能立刻知道最新的信息，无需重新训练。

- **解决“数据隐私”问题 (Privacy)**： 企业有很多私密数据（财务报表、技术文档）不能发到公网上训练模型。RAG 允许企业将数据存在本地数据库中，只在生成回答时临时调用，既利用了 AI 的能力，又保护了数据隐私。

### RAG 优化

我们将 RAG 的生命周期拆解为四个关键阶段：

**数据准备 -> 索引 -> 检索 -> 生成**

每个阶段都有对应的提分技巧。

### 1. 数据准备阶段：Garbage In, Garbage Out

数据处理是 RAG 的地基。如果切块（Chunking）切得不好，后面的模型再强也没用。

- ** 避免：单纯的固定长度切分**
- 例如：每 500 个字符切一段。这会导致一句话被切成两半，或者上下文丢失（比如代词“它”在这一段，指代的主语“iPhone”在上一段）。

- **语义切分 (Semantic Chunking)**
- 基于文档的**结构**来切分。比如利用 Markdown 的标题（# ##）、段落换行、或者代码块边界。
- **Overlapping（重叠窗口）**：在切分时，让前后两个块保留 10-20% 的重叠内容，保证上下文的连贯性。

- **元数据提取 (Metadata Extraction)**
- 不要只存文本！要提取元数据（文件名、页码、日期、作者、URL）。
- **作用**：在检索时进行**元数据过滤（Pre-filtering）**。
- _场景_：用户问“2023 年的财务数据”，你可以先过滤掉所有非 2023 年的文档，大大提高准确率。

---

### 2. 索引优化阶段：不仅是存文本

这一步的核心思想是：**“检索用的内容”和“给大模型看的内容”可以不一样。**

- **父子索引 (Parent-Child Indexing / Small-to-Big)**
- **原理**：
- 将文档切成极小的片段（Child，比如一句话）用来做向量索引。
- 当检索命中这个小片段时，不要只返回这一句话，而是返回它所属的**父文档块**（Parent，比如整个段落或那一页）。

- **为什么**：小片段语义更集中，容易被搜到；大父块上下文更完整，大模型回答更好。

- **假设性问题嵌入 (HyDE - Hypothetical Document Embeddings)**
- **原理**：用户的**问题**通常很短（“退货政策”），而**文档**很长且正式。两者的向量空间距离可能很远。
- **做法**：在存文档时，利用 LLM 为每个文档块**自动生成 3-5 个假设性问题**。将这些“问题”和文档一起存入。
- **效果**：当用户提问时，实际上是在用“问题”去匹配“假设问题”，匹配度会显著提高。

---

### 3. 检索阶段：准度提升的关键

向量数据库不是万能的，尤其是对于精确匹配。

- **混合检索 (Hybrid Search)**
- **痛点**：向量检索（Dense Retrieval）擅长理解语义，但经常搞错专有名词（如 SKU 编码、人名、缩写）。
- **解法**：同时运行两套搜索：

1. **向量搜索**（语义匹配）。
2. **关键字搜索**（BM25 算法，类似传统的搜索引擎，精确匹配词汇）。

- **融合**：使用 **RRF (Reciprocal Rank Fusion)** 算法将两边的结果合并排序。

- **查询重写 (Query Rewriting)**
- **痛点**：用户的提问往往是不完整的。
- _用户问_：“它怎么配置？”（系统不知道“它”是谁）。

- **解法**：在检索前，先调用一次 LLM，把包含历史记录的对话发给它，让它**重写一个独立的搜索查询**。
- _改写后_：“如何在 Linux 系统中配置 Nginx 服务器？” -> 再拿去检索。

---

### 4. 后处理阶段：重排序

- **重排序 (Reranking)**
- **原理**：向量数据库为了速度，使用的是近似搜索（ANN），它返回的 Top 10 结果里，可能第 1 名是不相关的，第 8 名才是最相关的。
- **做法**：

1. 先从向量库里捞出 Top 50 个结果（召回范围大一点）。
2. 使用一个专门的 **Cross-Encoder Rerank 模型**（如 BGE-Reranker, Cohere），精细计算“问题”和“这 50 个文档”的相关性分数。
3. 按分数重新排序，只取 Top 3-5 给大模型。

- **效果**：通常能带来 10%-20% 的准确率提升。

---

### 5. 评估体系：如何知道好坏？

不要凭感觉盲测，需要量化指标。

- **建立 RAG 评估框架 (RAG Evaluation)**
- 使用框架（如 **RAGAS** 或 **TruLens**）来自动评估三个维度：

1. **检索相关性 (Context Relevance)**：检索到的文档真的和问题有关吗？
2. **回答忠实度 (Faithfulness)**：AI 的回答是基于文档的，还是瞎编的？
3. **答案相关性 (Answer Relevance)**：AI 的回答解决用户的问题了吗？

---

### 📝 总结：一套标准的 Advanced RAG 流程

一个成熟的 RAG 请求流程应该是这样的：

1. **用户提问** ->
2. **Query Rewrite** (改写补全问题) ->
3. **Hybrid Search** (向量 + 关键词并行搜索 Top 50) ->
4. **Rerank** (重排序，精选 Top 5) ->
5. **Context Construction** (拼接 Prompt，使用父子索引取大块内容) ->
6. **LLM Generation** (生成答案)
